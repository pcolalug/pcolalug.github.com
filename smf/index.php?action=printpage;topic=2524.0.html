Title: Best way to update Slack
Post by: kilar on July 16, 2005, 07:02:05 am
I'm not trying to be elitist. What I am trying to point out is that there a hundreds of distros available, each with an intended use and audience to suit a multitude of personal preferences.

When you try to force one distro to act like another, it introduces an unknown element. For example, if you run emerde on Slackware, you will spend some time fixing broken apps because emerge was not designed for Slackware--it was designed for Gentoo. Slackware uses the System V init scheme, whereas Fedora, Ubuntu and many other distros use the BSD style. Gentoo uses an init scheme similar to System V, but uses proprietary tools to administer those scripts (rc-update). What all that means is that config files are not in the same place between certain distros, yet you are using a third-party app which circumvents the differences, but is not accurate in keeping track of those differences, thus resulting in a broken system.

If you use distro X, use its native package manager, or build from scratch. Or, know how to code, and fix the problems along the way, because there will be a lot of problems.

Slackware is a no-nonsense distro. One should administer it with the same philosophy. Otherwise, you are defeating the purpose of the distro's philosophy, and using the philsophy of another distro. Use what you want--there are hundred distros available. Just don't complain if you take distro X, try to make it work like distro Y, then complain about it not working.

Quote from: kilar

Why does every freaking distro need it's own package manager?! I'll never understand the reinvent-the-wheel mentality of the OSS community. The "Linux gods" is an absolutely ridiculous thing to say. Anyone can make a package for whatever platform/distro/package management system they want and release it. What breaks a package management system is the quality of the port and the quality of the packages created for that package manager. Debian has the right idea with their packages because so much testing goes into them before they get put into stable. You know those packages are going to work, period. If you run sid, you may get some broken packages, but it's a fast moving repository, so it will probably be fixed quickly.

Now, the big problems I see with running a non-distro-native package manager is first, the quality of the packages isn't always the best. Some body out there gets a wild hair to package up something and builds the package with little understanding of that package format. This gets released with the assumption "Hey, it works on my Amiga! It should work on everyone's system." No testing! Then that author loses interest and you have a broken package sitting in the repositories. People complain, a few fix it, but no one takes up the responsibility of maintaining the package.

Second is dependent on how active the community is surrounding that package management system on that distribution. For example, Apt on SuSE is extremely active. If there is something that won't install or is broken on install, more than likely, it will be fixed within a day. Mind you, I run on the bleeding edge with a lot of Alpha and Beta releases. I couldn't see apt-rpm being very popular for Gentoo...although I'm sure it exists. The more people involved, the more testing and the more submissions to the repositories. You have more quality in the user pool. If there aren't many people involved...well, see my first point.

So there you go. My view on package managers. It all depends on how many people are involved for the manager/distro/architecture which will help with the testing of the packages leading to better quality and less broken systems. Linux gods have nothing to do with it.
