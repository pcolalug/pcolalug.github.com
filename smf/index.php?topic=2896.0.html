I don't know if I'm supposed to do this, but I'm going to share what I've found so far.

OpenSSH 4.3 is running on port 2022. Yep, that's it. Also, it appears to be a recent release of OpenBSD. Short of brute force dictionary attack, there is little risk of getting into this box. I am fairly sure pf is running because openssh is the only port that shows as open and ident (113) is closed, rather than dropped. This helps for services that check ident; rather than waiting for a request to timeout, the requesting service gets a closed back and moves along. Mostly ircds and ftpds do these checks.

Anyhow, I plan on starting a dictionary attack, but I am going to skip the obvious root account, as I'm sure it is denied login. Since I don't know a username, I'm going to just go with some obvious ones like bhoff, seawolf, lloydr, slackwolf, bhose and such. I'll let you know if I notice any kind of limiting as I will get my dictionary attack going with Hydra. I highly doubt this will work. Typically, dictionary attacks are only good against weak passwords on known accounts (root, administrator, guest, ftp, etc). To be honest, I wouldn't be surprised if there were only one or two users on that system that have been given ssh permits. The chances of finding those are very slim. Normally, you would want to exploit a vulnerable service or website and gain root level access to either obtain the passwd/shadow files or create a new user for yourself. Unless there is some unpublished vulnerability in the OpenSSH server, that is not possible.

I don't have time to tinker with this, but if I did, here's how I'd attack the problem:

Don't look directly at the box you want to attack - of course its locked down completely from the outside, otherwise this would be a short contest. I would start by sniffing the entire slackwolf.com subnet and find a vulnerable server, and literally attack it from behind. Just a quickie glance I found DNS servers on 66.80.196.243 & .244, a mail server on .245 and I'm sure there's an exploit for those somewhere. (Lloyd, you're brave for doing this on you own network) The idea is to break into the box indirectly through a service/IP/port that it trusts to be safe from within. (As a bonus hint, look for exploits that use IRC, since we know Lloyd uses IRC, there's probably not a rule blocking outbound IRC (6667) traffic. Once you have a hole via IRC outbound, chances are the firewall will keep state, and then let you inject directly through the firewall via port 6667)

From the black hat perspective, breaking into a firewall nets you very little in comparison to breaking into the systems that the firewall was meant to protect.

For a future contest, what you should do is setup a typical office setting (for me, that's an OpenBSD firewall with Linux and/or Windows servers behind it, and allowing typical services such as ssh, remote desktop, web, mail, dns, etc). Put a file on one of the servers named: "content_winner_read_me" with some code in it so whoever discloses the contents of the file first wins.
