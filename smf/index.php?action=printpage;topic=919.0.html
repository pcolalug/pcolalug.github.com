Title: Red Hat 9: If I ever get it installed I'll tell you about it
Post by: imnes on August 05, 2003, 06:52:04 am
If you're having that big a problem installing and your CD/DVD's check out OK then most likely you have a hardware problem on your machine or some sort of incompatibility with the installer's kernel.  

You can try disabling your secondary cache in your bios, disabling ACPI, disable uDMA access on the drives, whatever.  Point is the installation process is very capable of killing an unstable system.  

My little web server (which was originally a 75mhz pentium now magically running a 450mhz AMD chip, I still haven't figured that out seeing as how the manufacturer says it doesn't support 100mhz bus) would reboot mid install, everytime, on every distribution.  I finally disabled all the cache in the bios and it installed perfectly.  After installation I re-enabled the cache and everything is happy.

Or try another distribution.  Maybe I recommend slackware?  If you want to learn linux, then slackware / gentoo / some other clean implementation are the only way to go.  If you install redhat you're only going to learn how redhat's configuration tools work, and you'll probably be lost on most other systems.  Not a bad thing if you just wanna use linux, but if you truely wanna learn about UNIX like systems, there really is a better way :)

Quote from: Rugmonster

Perhaps you've never seen Mandrake's URPMI in action...  It solves dependencies automagically.  Although I've not used Debian, I believe it's pretty much like Deb's apt-get.  All you have to do (at a command line) is type "urpmi <packagename>" and it's downloaded and installed.  Very very simple and effective.

I'm not sure it's completely accurate to say someone doesn't "know Linux" because they use a GUI/RPM based system.  As you know, the CLI is right there under the surface.  I do agree that much of the power of Linux is in the CLI, but there are some things that *really are* easier to do in the GUI.  It's not a bad reflection on someone for using it, and doesn't necessarily mean they can't do it the CLI way.  In fact it could be argued that knowing both ways expands your knowledge of Linux in general.  I use Mandrake mostly, but also have a working Slackware box that boots to the CLI.  I find it challenging to learn the differences between these distros, and enjoy both of them.

Anyway that's my $.02 worth...

Post by: rugmonster on August 05, 2003, 03:51:06 pm
Yes, apt-get does work for rpm's...but it isn't something that is configured and used by default. Yes, there are wonderful *GUI* tools out there, but to use straight RPM, the Redhat Package Manager, you run into dependency hell and as far as I know, apt-get is not included or configured in a default Redhat setup...let alone a minimal one. Also, is there a large repository of packages like Debian or Gentoo has for rpms? I know there is rpmfind, but I've had problems find stuff there that was available in Debian's repositories. Apt doesn't work unless you tell it where to look.

And one more thing, has anyone actually tried to make an rpm or deb? deb's are so much nicer and supposedly gentoo's packages are supposed to be even nicer than debs.

Hooray for crappy package management! Yeah, Windows beats Linux out there. Can you click Next?

Finally, I tell people to learn the CLI way of things because you won't necessarily have those pretty GUIs on other systems. If you ever plan on administering another flavor of unix or systems for a company, you more than likely will not have those tools, but the CLI and config files will be there. Believe it or not, a lot of these packages we use on Linux are ported to other systems. What do you do when you have a system 3000 miles away and all you can do is SSH in? Unless you have bandwidth and system to spare, tunneling X isn't always the best option.

My $50.02.
