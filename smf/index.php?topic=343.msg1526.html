I've never had to set environment variables, but it depends on what shell you're using.  If you're in bash, (you probably are), then edit something like ~/.bash_profile or maybe even ~/.bashrc in your user directory.

The startup scripts in RedHat (and other derived distros like Mandrake) are really best dealt with through the configuration utilities.  If you just want to enable / disable different apps from running, use chkconfig.  It can be used to list which runlevels an app starts in, and to enable/disable them.  

Runlevels are just used to classify what mode your system is operating in.  Kinda like windows has normal mode, safe mode, command prompt only, etc.  To see what the different runlevels for your system are, you'll need to look in the file /etc/inittab, your system is probably the same as mine, in which case:
Code:

   0 - halt (Your system is in runlevel 0 when it is shut down)
   1 - Single user mode
   2 - Multiuser, without NFS (The same as 3, if you do not have networking)
   3 - Full multiuser mode
   4 - unused
   5 - X11
   6 - reboot (Your system goes to runlevel 6 when you are rebooting it).


Here's an example of GPM:

GPM is the program that gives you a mouse in text-only mode.  To see which runlevels it's operating in, you would type:
/sbin/chkconfig --list gpm

When I type that, I get:
Code:

gpm             0:off   1:off   2:on    3:on    4:on    5:on    6:off



What this tells you, is that GPM will be started when your computer enters run levels 2, 3, 4, and 5.  

To modify which runlevels it starts in, (for example, I want to make it start in 6 also), you would run:
Code:

/sbin/chkconfig --level 6 gpm on


To turn it back off for runlevel 6, do:
Code:

/sbin/chkconfig --level 6 gpm off


And also, there are lots of useful scripts in the directory /etc/rc.d/init.d, my system has:
Code:

alsasound  firstboot  isdn       netfs    pcmcia      saslauthd  syslog
anacron    functions  kdcrotate  network  portmap     sendmail   winbind
apmd       gpm        keytable   nfs      postfix     single     xfs
atd        halt       killall    nfslock  random      snmpd      xinetd
autofs     iptables   kudzu      nscd     rawdevices  snmptrapd  ypbind
crond      irda       lpd        ntpd     rhnsd       sshd



If you run these scripts without any arguments, they will tell you how to use them.  These are the actual scripts that are called while your system is booting up / shutting down, etc.  Take the gpm script for example, if I want to see what arguments the gpm script accepts, I just type:
Code:

etc/rc.d/init.d/gpm


And it tells me:
Code:

Usage: /etc/rc.d/init.d/gpm {start|stop|restart|condrestart|status}


So, if I want to see the current status of the program, I can type
Code:

/etc/rc.d/init.d/gpm status


Which on my system tells me:
Code:

gpm (pid 832) is running...


Which means the program gpm is running under program ID (pid) 832.  If I want to close the GPM application, I would issue:
Code:

/etc/rc.d/init.d/gpm stop


Since I'm a standard user and not root (you need to be root to do most stuff in those scripts), I get an error message:
Code:

/etc/rc.d/init.d/gpm: line 180: /var/run/gpm.pid: Permission denied
/etc/rc.d/init.d/gpm: line 196: kill: (832) - Operation not permitted
/etc/rc.d/init.d/gpm: line 201: kill: (832) - Operation not permitted
rm: cannot remove `/var/run/gpm.pid': Permission denied    [FAILED]


But if I were root the program would terminate.  Typing
Code:

/etc/rc.d/init.d/gpm start

would of course restart the program.

The restart command is sort-of a shortcut, it calls STOP and START for you.

Anyway, you can look through them and play around if you want.  The only one I ever use is the network script if my router reboots and I need to get a new IP address or something, I'll do it now so you can see what it looks like.  This command takes down my ethernet connection, unloads the module, then reloads the module and contacts the DHCP server for a new IP address.
Code:

[root@localhost imnes]# /etc/rc.d/init.d/network
Usage: /etc/rc.d/init.d/network {start|stop|restart|reload|status}
[root@localhost imnes]# /etc/rc.d/init.d/network reload
Shutting down interface eth0:                              [  OK  ]
Shutting down loopback interface:                          [  OK  ]
Setting network parameters:                                [  OK  ]
Bringing up loopback interface:                            [  OK  ]
Bringing up interface eth0:                                [  OK  ]
[root@localhost imnes]#


That should be enough to get you started.  If you have more questions, post 'em.

I personally do things a little weird with my system. Technically, executables should be placed under a bin directory. /bin is used for the programs that are necessary for bare minimum use (ls, cp, rm, etc.). In the old days of Unix, the extra programs, not necessary for minimum operation, went into /usr/bin. If you notice, most of the directories under root are also under /usr. It was and to some extent still is common practice to have your root be its own partition and then you mount usr, var, etc. This is so if one of those partitions gets damaged, you only have to worry about rebuilding one part of a system rather than the whole thing. Anyway, /usr/bin is now where the majority of your packages are installed. It is recommended that any other programs that are installed be installed under /usr/local with /usr/local/bin being where you put your executables.

What I do weird is I take all my packages that I install from source and put them in their own directory under /usr/local/pkgs/<pkg name>. I then symlink the appropriate files to their respective directories under /usr/local. For phoenix, I unpacked it under /usr/local/pkgs and then made a symlink of /usr/local/pkgs/phoenix/phoenix to /usr/local/bin/phoenix. I found that if I made this symlink, the phoenix binary would then go on its own to open up the other files it needs. It works out quite nicely and I can update phoenix by writing over one directory. Same with my source packages as well.

One of the most confusing things to me still is the startup process, and what scripts are run, and how environment variables get


Okay...in most Linux distros, it uses the SysVInit (as in sys 5) style of scripts. You typically have /etc/rc.d and a directory either /etc/rc.d/init.d or /etc/init.d. The init.d directory has all of your scripts which have the options of start, stop, reload (optional) and status (optional) that either start, stop, reload or give you status on your various system and server daemons. You then will probably notice under /etc/rc.d a number of directories rc(number).d. These all correspond with the different runlevels and then under there are more scripts that have very similar names to those scripts under init.d. A runlevel is the mode at which the system is run at. Here is a list of some of the run levels (there can be more...):
  • 0: halt the computer
  • 1: single-user mode
  • 2: multi-user mode without networking
  • 3: multi-user mode with networking
  • 4: reserved for customization, otherwise does the same as 3
  • 5: same as 4, it is usually used for GUI login (like X's xdm or KDE's kdm)
  • 6: reboot the computer
    • [/list:u]
    Now within the rc(#).d directories have symlinks to the scripts in init.d. The numbers on those symlinks refer to the order of execution for that run level. They are all excuted one at a time, but the numbers dictate in which order they are executed. This is important if say you are going into runlevel 3 and are mounting your foreign Samba shares and need your networking up to do so. Obviously, you need your networking established to mount those directories. Also, you will notice a K or S and these typically refer to Kill or Start, but if you are under runlevel 0 or 6, S stops the process since the system is going down.

    I hope that explains what you want to know. Any other questions, please feel free to post.
