23  Misc / Computer General / Re: Unix Load Averages Explained on: May 21, 2006, 04:13:39 pm
load is some sort of combination of disk IO, ram utilization, swap utilization and cpu.  i'll read the article for sure, cause that's about all i know about it:).

i've had boxes with 0% cpu and high loads (ide servers sending lots of files, disk IO problems -OR- boxes without enough RAM ).  I don't like loads hanging above 2 regularly.  That's a bit much for me.  proper performance (imo) is below 1 under normal usage.

if you haven't tried it, run vmstat.  it shows disk IO and swap usage.  great command to add to your regular box checks.  get familiar with your box's disk IO and swap usage . .makes ALL the difference.
24  Distributions / Security / Horde/Imp on: May 21, 2006, 04:07:54 pm
Hi everyone.  I introduced myself in the Introduction forum and rugmonster asked that I post about the hacked boxes I've had to handle over the years.

The latest hack was actually only a few weeks ago.  I do consulting/admin work for a cable/internet company in south florida (all remote).  We currently use horde/imp for webmail for customers (www.horde.org).  It's a great product and has served us well, however, it was the source of a server breech recently.

A version of horde experienced some sort of /tmp vulnerability via apache.  The perp could only transfer files only to directories writable by apache and could only run programs AS apache, and this guy stuck in code for an irc bot (script kiddie). 

The REASON we were hacked was a lack of upgrading of horde.  We run Redhat Enterprise and everything was updated except horde.  It's a rather long laborius process that takes a few hours and I am only contracte for five hours/week.  Without making excuses, I didn't get to the upgrade until the breech occurred.  I was able to verify a clean box after cleaning the irc bot and upgrading horde/imp, etc.  I think the fix was released by the horde group in march and I did the upgrade mid or end of April.

It was normal scheduled port scanning that caught the bot, which was only active for just over a day or so.  No REAL damage done, except to my pride.  I haven't had a box compromised in five years.  The previous compromise was a full rootkit on a box that I thought was offline, but was turned back on by a tech would thought it wasn't supposed to be off.  My monitoring has improved since then (arpwatch, etc), so that kind of thing is not as likely (never say impossible).
