Goals of the project would be:

Phase 1:
Write the base code in Perl and run as a daemon. No gui, just email notification. One of biggest problems I have with What's Up is that if you are monitoring the email server, and it goes down, how's what's up going to send you the message? To make things easy, I see a configuration file thats very similar to Apache's VirtualHost, where you can identify the server address, what services to monitor, and who to email (using the perl smtp mailer I've written) when they fail. It would also create a log file for reporting purposes.

Phase 2:
For those that just gotta have a gui - Write a QT front end (QT so you can cross platform it) that will provide the visual part of the perl daemon - the perl daemon would still be running, and the GUI would just monitor the log file to determine the current state of each monitored machine.


Am I totally without my marbles, or does this sound like a do-able idea? I emailed the current admin of the project, but he/she hasn't responded yet.

I was looking at some remote monitoring stuff done several years ago.  Some cron jobs would periodically run that would try to go out and hit various servers and services on your network.  If something wasn't available it got flagged and rechecked in a few minutes.  If it failed again then it got flagged as down.  Another script periodically ran and would check the list of failed services, if it was a non-vital service then an email would go out to the support staff for that service.  If it was a vital service then a script would use the system's modem to dial out and page the team member responsible for that service.

I don't know if this relates really to what you're going to build or not?
