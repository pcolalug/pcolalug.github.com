The initial install was quick.  Building the base system takes a while, I just let it run while I was at school and it was done by the time I got back a few hours later.

Then you have to compile a custom kernel and other misc tasks, that took me all of about 20 minutes.  

The hard part is waiting for X and your window managers and other apps to compile.  I let X and Gnome2 compile overnight so they took less than 8 hours.  Now I'm building the apps I need as I go, got galeon, blackdown java, gimp, open office is gonna take a while.  I tried to compile gaim but it refused, stopped on some file, an Icon I think but I haven't looked into fixing it.  I installed CenterICQ which is a really nice console based instant messaging program I used in BSD when I wasn't in X, it supports AIM, ICQ, and some of the other chat clients all in one and integrates them with each other, very nice if you want to check it out.

As it sits right now I have a functional system with very few apps, but it's d**n fast.  So far I've spent about 3 days on it.  I would guess it would take me a week of playing with it part time to get it to the same state as my typical RedHat / MDK install.

Quote from: rburt3
And my problem has always been when to module and when to compile in...


My view on it is...always compile in. True, you might not always be able to put the kernel on a floppy to boot the system, but a floppy kernel doesn't need to be preemptible or have sound drivers or smbfs kernel support. For some reason, I just don't like modules. I like to have everything compiled in. I leave module support in, just in case I need to swap a nic and need to insmod something later (happened once), but I usually compile something in unless I don't know what the bugger is and if I'm afraid I might need to pull it out to make the system stable again...but that is why you always keep your last good kernel available to boot in case that new kernel flops.
