I've stayed out of this topic because I don't normally deal with the Servers. However, I have helped our engineers choose Linux servers for projects they've been working on. Usually when they ask what the best server to run is, I answer by asking what programs they plan on running. I then send them to the programs web sites to see what distros those programs are being developped on. In the case of Apache, Samba, Sendmail and other common server apps, you can find them on most server distros. However, if you're going to be running something new/out of the norm, find out what it's being developped on and that would be the way I'd recommend to go. You're going to get better compatability, stability, and support that way.

This also means if the programs are still being developped and tested on RedHat 7.x then stick with it.

I've found that most big server programs are normally at least one distro version behind the current version. So, I wouldn't necessarily plan on going with the latest and greatest version.

Quote from: Rugmonster
I hate to ask this, but why would you guys recommend Slackware?

Honestly, I would suggest using whatever distro you feel comfortable with since it is an individual issue. I have never thought of Slackware as being a big server distro, but I'm sure it does fine. I have use Slackware on small servers, but eventually moved to Redhat as the needs of those servers grew as it was easier to admin those. I think Debian would be a better choice due to it's more robust package management and rock solid stability (which I have seen lacking in Slackware comparitively speaking).

To me there is no "Best", rather what fits your needs and wants better than the other options.


I'd recommend slackware because for me minus installing an nVidia graphics card (which I think is more the fault of the computer's builder) it has been rock solid for me.  Even before I knew about swaret I found no difficulty in using the distribution at all, even without X.

After using Fedora I think Slack is a huge improvement in speed.  With Fedora it took MINUTES to do things whilst in Slack it's no problem at all.  Also I discovered that Slackware wasn't nearly as hard to install (and reinstall) as people made it out to be (to me, anyway).  After understanding the basics of the installation I didn't think it was hard at all and could've done it myself.

After trying to install Debian for three days and ending up with a half-arsed installation I wouldn't put it on any other machine.  Just from my experience that was the worst thing I've tried to use.  During the installation process it had to apt-get everything which would normally sound like a fine and dandy idea...until the horrible realization that if it failed on ONE package I would either have to totally skip it (a bad thing for some essential packages) or I could abort the ENTIRE installation.  Whatever happened to, "Just try to get it again?"

Plus I hate to say it but I think Slack's installer was friendlier which is saying a lot.  As a developer I've never had an issue with Slack, it came with all sorts of wonderful development packages (even Ruby with a little coaxing).  And while my server duties were minimal it handled them easily.




Oh, and I'm a big Slackware fanboy of course. I'll admit it.
