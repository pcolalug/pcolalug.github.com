Quote from: dpeach
...but I have read that they are specialty things and that most applications can get by without them.

Not once you've used triggers and stored procedures, they're not! Take the "cascading delete" feature of PostgreSQL. Say I have a 1:n relation in which the items on the n side have no meaning without the 1 side. If I turn cascading delete on the fk in the n table, if I delete a record from the 1 table, the associated n records are automagically deleted. It also allows things like updating tables via a view. Very handy indeed.

Quote from: dpeach
MySQL is easier to learn and (almost) as powerful.

IMHO, MySQL isn't nearly as "powerful" as PostgreSQL. Not by a long shot. No offense, and lets not turn this into a "my RDBMS is better than your RDBMS" match, but the capabilities of a mature RDBMS like Postgres or Oracle *far* exceed the capabilities of MySQL. Not that this makes them "better", per-se, just more capable for building data-driven applications.

OK. I did about 5 minutes research in Google and here's what I found.

1. MySQL has been around longer as a fully functional development Database System. Postgres has just come out of the "Academic Research" arena in the last 5 years or so giving MySQL time to become more of a standard backend in the Linux community.

2. There were some tests run back on 2000/2001 by PC Magazine where they got together the designers of Oracle, MSSQL, MySQL, PostgreSQL, and a couple of others. MySQL blew the dorrs off the other databases in speed and stability.

Granted, things may have changed in the last few years, but that doesn't change the fact that MySQL had time to be accepted as a standard.

That's why it is so popular.

Why is MySQL so popular?
« Reply #8 on: June 23, 2003, 10:12:09 pm »
I don't really know MS Access, but does MySQL offer any advantages over that or are they basically comparable?

If you're working on a small one-man project, then something like MySQL is ok.  But if you're working on a project where several programmers are writing code to work with a database then you really want stored procedures and triggers to safe-guard things.  For example, if you had two tables, one containing user profiles and another containing user privilages, you might want to have the database automatically set some default privilages to new users when they are added to the profiles table.  You could leave that up to the programmers to code in their scripts themselves (and hope they all remember to do it all the time), or just let the database itself handle it.  

PostGres is a lot more powerful and is more of a true DBMS than MySQL.  And, at least from my limited exposure, the code to use either is nearly identical.  So that's why I was surprised to see MySQL having a much larger (well, at least apparently larger) number of people using it in their websites and applications.

In reality you can code around most of the lacking features of MySQL, but why do that if you don't have to?  Unless you just like writing more code, like some of us do

But, maybe it's convenience.  Most linux distros come with MySQL installed and ready to go.

MySQL vs. PostgreSQL
« Reply #12 on: July 01, 2003, 02:49:25 am »
From a database programmer point-of-view, you don't need stored procedures and triggers to do atomic operations.  MySQL is quite a bit faster that PostgreSQL, it uses standard SQL to do DDL (PostgreSQL is an absolute *nightmare* in this respect!), and the MySQL crowd aren't as snobby

On the other hand, sometimes you need stored procedures and triggers, but it's not very often - people who want to brag about the speed of stored procedures need to re-run their query against MySQL - stored procedures don't give you a bit of speed increase.  Triggers?  They can be coded for - triggers are for lazy programmers
