I don't know if anyone has had any experience with this, but I figured I'd throw it out there. Let me first say, I know it isn't related to linux in any way, but I put it here for that reason.

The situation:

My friend and I each have Linksys WRT54G routers and a Windows 2003 Standard Edition domain controller. We want to setup a VPN between our networks so that we can setup a trust between our domains and utilize each other's network resources to include SSO and such through domain trusts.

Here is a simple diagram:


The problem:

We looked into utilizing the WRT54Gs as vpn server/client boxes with the Sveasoft firmware; however, it only provides server capabilities requiring the client side to have a VPN dialer on each system.

Windows Server 2003 provides for Router to Router VPN connections, but we haven't been able to figure out how to get this working. We can get connections up, but he can't route back to my network past my 2k3 server and I can't get past his 2k3 server. Basically, I can ping his VPN interface and the actual interface from my 2k3 server and he can do the same from his side, but systems inside our network can't ping to VPN interface on the distant end or to the distant end network. With that said, the default route for the workstations are set to the WRT54G and that has routes for the VPN network and the distant end network set to point to the 2k3 server. The 2k3 server has static routes for the distant end network to use the demand dial VPN interface.

It's as if the 2k3 boxes aren't routing traffic across the virtual VPN interface and we're somewhat stumped. I would appreciate it if anyone has an idea of how this is supposed to be setup to give me a clue. I feel as though I'm missing something very minor. If you have time, a complete walk through would be much appreciated.

If anyone has an alternative, I'd be more than willing to hear it. I was thinking that I could get him to use his router as a PPTP server and then use my linux box as a client and then route through that instead since I know how to make a linux box route.

That article follows every other guide MS has put out...the router has two nics. This shouldn't be necessary because the VPN connection is a "mini-port" which should be a virtual interface (at least in my idealistic Linux world).

As much as I hate to concede defeat, I'll accept it in this case. Last night, I compiled a new kernel with PPP support. I plan to use PPTPClient on my linux box to connect to a PPTP server running on my friend's WRT54G running Sveasoft's latest public firmware. When PPTPClient connects, it makes a new virtual PPP interface, to which I can route traffic between. There are few examples I can think of where something is easier on Linux, but this will definitely get appended to the list if it does work.
