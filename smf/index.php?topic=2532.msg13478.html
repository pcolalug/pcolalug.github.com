Supposedly GTK (and MS Windows) native widget rendering is in the current 1.6 Java builds.  I tested the latest out with various themes that are included with Dropline Gnome and it worked fine. 

JTable has been enhanced, tables now have sortable columns. 

Layout managers now look at the platform that the app is running on for visual hints about things like spacing between components, this helps the app integrate better on your platform.

(I tested the QT-GTK -- or is it GTK-QT -- engine, and it currently does not work,  I've started looking into this but not sure if it's possible to use because of the way QT themes work.)

New drag-and-drop system to work like users would expect.

Double-Buffering of the GUI so window manager updates are faster and smooth.  (No more 'gray rectangle' when something moves over a java app and it's waiting to redraw).

Printing enhancements.

Can put any components you like in tabs of a Tabbed Pane.

Java2D updates so that java apps now offer the same text rendering capabilities as the supported platforms.  (Including antialiasing for Windows and GTK).

System Tray API for integrating icons, menus, notifications into the system tray / task panel / notification area of the supported platforms.

Enhancements to Web Start deployment so that apps can install over the web and integrate into your Start Menu (or Gnome Menu), desktop shortcuts, etc.

Simplified JRE installer for end-users.

https://mustang.dev.java.net/

Quote
A.) How much of the underlying code for the App has to change for this to work?


Basically none.  Most of the changes are happening within the implementation of Swing without changing the API.  One thing that does require additional work is table sorting.  By default, the behavior is like previous Swing releases (probably to maintain compatibility).  Table sorting is disabled.  To enable sorting you have to implement a method that basically tells swing how to sort the objects in your table.  If you've used java you're probably familiar with this using Collections type sortable classes.

Quote
B.)  How fast is the UI?
 

This is pretty subjective.  The reason most swing apps (made on Java 1.3 or later) seem slow is because the programmers writinig those apps didn't use threads correctly.  Java Swing does everything in a single thread (I think QT, GTK, Windows, and Mac all do this too).  So if you have a method / function that gets run when the user clicks a button on your GUI, the GUI will appear to freeze while that function is run.  This is fine for small instantaneous tasks.  For longer tasks the programmer should run that code in a seperate thread so that Swing can continue to update / draw the GUI while it's happening.  That's the reason swing usually feels slow (if I understand it correctly). 

I haven't dug through the code too deeply so I don't know how the drawing is actually implemented, if windows is drawing to the java app directly, or if the java app is telling windows to render a button off-screen and then java draws it to the app - I'm really not sure.

As far as applets, all changes that are happening in swing should also be available in from applets.

On another front, Sun has finally made some really nice advancements in NetBeans, and one of the deriviative products Sun Java Studio Creator.  And both are now being made available for free.  With the new GUI editor in NetBeans, Swing development is approaching the ease of Visual Basic.  And similarly with Studio Creator you can drag+drop web applications easily.  It's pretty impressive stuff but both products (version 5 of NetBeans, and 2 of Java Studio Creator) are still in Beta, so I wouldn't do any production projects in them yet, but they are certainly good for testing.  Just make sure to hit the Save button often
