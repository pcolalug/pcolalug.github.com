The ./configure (or I think just configure), make, make install routine is if you are installing from source. If, you need to install something from source, then that is the way to do it. If you are installing an RPM, then that ain't it.

First I would go with Yast (I think that is what the installer is called for Suse). Try installing from there. If it does not have what you need, but you can get your hands on an RPM of the program you want, then download it and proceed 2 different ways. One is to try double clicking on the RPM. It may bring up your installer (and probably ask for your Root password). This is the way I do it. If you need to install from the command line, then someone else will have to help.

If, on the other hand, the program you are trying to run does not actually have to be installed or it comes with it's own installer, it should be documented on the site or in the tarball that you downloaded. Some will have something like "programname.sh". If it has its own installer then you can usually type "sh ./programname.sh" from the directory that contains the program.

If it is something that will just run when you unzip the folder, then find the program file and type "./programfilename" and enter. Cross your fingers and hope it starts.

These are the ways that I know how to do it.

What program are you wanting to install? Maybe we can give some specific instructions.

BTW, the "./" means "look in the current directory for the file, not in the path".

With an RPM based distro like Suse, it is generally a good idea to install only RPM's, instead of mixing in manually installed stuff.  This is true even for non-newbie people, and more so for newbie people...

To compile from source (install manually), the general procedure is to unpack the source code tarball (something named like "program.tar.gz") into a temporary directory.  Then you need to change directory to that source code tree (where you just unpacked it to), and type:  "./configure".  After that completes, type "make".  Then do the "su root" thing and type "make install".  That process will work for most packages.  Something to always look for is a file called "README" or "INSTALL" in the source code directory that you unpacked.

As for your last question, yes it is a BAD idea to log in as root.  Always use the "su" command to get root access for a while IF you need it, and then "exit" back to your normal user account when you're done with whatever you're doing as root.  This will prevent you from accidently destroying your entire system, which is quite easy to do as root.

Again, I would highly recommend that you only install new software in the RPM format, using the graphical tools (YAST) provided by Suse, at least until you get more comfortable and knowledgeable.

The reason ./configure returns a "bash: ./configure: No such file or directory" is that there is not a command that is ./configure. You are running a program within that directory that you untarred. So only if there is a configure program within that tarball will it work.

You cannot just run "./configure" and have it spew out your options.

Quote from: bhoff
Well actually if you are running a properly secure box ./configure should only be acesible to Root. To keep people from installing malious apps.


Brett, ./configure doesn't *install* anything.  It configures the Makefile according to your hardware.  Then the "make" step compiles the application.  Then (and only then), the "make install" step *installs* the application.

That's why a person needs to be root to *INSTALL* the application, because the default installation locations are usually writeable only by root (although the location can usually be overridden with command line prefixes).  However, anyone can *configure* and *compile* the software in their home directory.  Grab a tarball for an app and try it yourself, as a normal user...
