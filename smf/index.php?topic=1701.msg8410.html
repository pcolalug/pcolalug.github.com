OK, so is It FrontPage's responsibility to ONLY be able to publish 100% compliant documents or is it the authors responsibility to make sure he/she is only publishing compliant documents. I think it's the later and I'll explain why.

MS makes FrontPage and IE. IE, in addition to being a standard web browser has features designed to work with other MS products (Windows, Office, etc.) FrontPage, being a MS product, should also have the capabilities to take advantage of the IE extra features. If I'm publishing for an internal network in a 100% Microsoft company then I may want to take advantage of some of those features. However, If I'm publishing for the general internet then I'm going to want to make sure that "MY" pages are 100% compatible with standard web browsers.

The bottom line is that YES FrontPage CAN publish 100% compatible code. It's up to the author to make sure they aren't breaking any compatibility with their finished product. That's why you should always check your pages with at least two or more different Web Browsers.

I've published pages with FrontPage and by default the pages have been compatible with Mozilla and Galleon. You would have to go out of your way (and IMHO know what you're doing) to make the pages not compatible with other browsers.

Quote from: steve
OK, so is It FrontPage's responsibility to ONLY be able to publish 100% compliant documents or is it the authors responsibility to make sure he/she is only publishing compliant documents. I think it's the later and I'll explain why.

I agree with you. It is not FrontPage's responsibility to produce perfect code. I think they should and could do better. But it is the responsibility of the author to make sure that it is.

Quote from: steve further
The bottom line is that YES FrontPage CAN publish 100% compatible code. It's up to the author to make sure they aren't breaking any compatibility with their finished product.

I have to disagree with that. FP maybe can if you clean up the code before you hit the save button. But to say thay FP does is quite a stretch in my understanding.

Quote from: and again steve
I've published pages with FrontPage and by default the pages have been compatible with Mozilla and Galleon. You would have to go out of your way (and IMHO know what you're doing) to make the pages not compatible with other browsers.

It would not be hard to make code that did not work in other browsers. The fact that it worked in a compliant browser is coincidental.

I am not saying that everything produced by FP is horrible. And, yes, the author can do some things to make the code better or worse. My point is that FP does not produce good clean code.

You can have poor code that shows up something in all the browsers, but it probably won't all be displayed the same in all of them. If your code is compliant, it should look very much the same in all the browsers (not identical, but close). I personally had a stylesheet in my page that looked great in some browsers, but not in Firefox. The problem was something small (don't even remember now). But that change made it view the way I wanted in all the browsers I checked it with.

To IE's credit, it has a way of taking poorly written code and guessing at what the author meant. Many times it will display fine in IE, but not in other browsers. It is because IE guessed right. That is wonderful, but it is not helping people to learn to write good code.

I am not sure what has gone wrong with the page in question. But, it does not work in browsers other than IE (I am assuming this is a true statement since no one has told us that it does not work in IE). And since these other browsers are supposedly compliant then one of two things has happened. Either FP created some wicked code, or the author did something to intentionally keep other browsers from using his page. I am going to venture a guess here, but it seems that from the quality of the page and content that I can see, the author probably would not know how to intentionally do this.

Looking at the output from the validation site tells me that this page is not compliant. Because it does not have that doctype that kicks it out of compliancy. Unless our author pulled that out, then we have proof positive that FP did not write 100% compliant code. Small thing? Maybe, but we have no way of knowing what it is supposed to be until that is defined according to the W3C standards. Again to IEs credit, it tries to figure it out, the other browsers say that they cannot render the page as intended because they follow the rules.

Again, I am not here to say that FP is wicked. I am just attempting to work out whether it is true that FP does not produce good clean code. I think it is obvious that it does not.

Therefore, I would have to say that FP is not as good of quality as hand coding. But, that using FP is fine if what you are looking for is a quick and easy solution to getting a page up quickly. On the other hand, if you think that FP is the "be all and end all" (whatever that means) of web programming, then maybe you need to rethink your standards of what is good and bad code.

I feel that some of this is coming off strong and that it is not coming accross with the lightness that I am feeling. So forgive me for that. I am just in teaching mode this week.

If I have said anything that is untrue, then please do not hesitate to correct me. I am trying to work from facts not opinions. If I had FP I could generate a page for you and be able to pick apart the code and the problems, as long as it is HTML and not another language. I also do not claim to be a real authority on web programming as a whole, I just understand some HTML and like to work within rules and standards.
