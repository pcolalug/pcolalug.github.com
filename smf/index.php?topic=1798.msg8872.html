Another good tip in the same vain as -p or --pretend is -a. This tells portage to calculate dependencies for what you want to do, then asks you if you want to perform the emerge. Its a little faster, since portage only has to calculate dependencies once. So, to see what you would get when you want to udate your system, just do a
Code:
emerge -Ua world
If everything looks good, all you have to do is answer 'y' when it asks.

For a more thourough update, use
Code:
emerge -Ua --deep world
The --deep flag tells portage to also examine the dependencies of the packages it would update.

Finally, if you change USE flags, compile options, timezone, or just want to be without a computer for several hours, try
Code:
emerge --emptytree --deep world
(the --deep may not be needed here, I can't remember). This tells portage to re-emerge every package in the world file as if it had never been there. It basically re-compiles all the software on your system (gcc, glib, the works). This has come in handy once when I screwed up glibc. I basically booted from the install cd, chrooted into my system, unpacked the stage 1 tarball back onto my system, sync'd my portage tree, then ran the above command. Fresh clean system in 24 hours .

Quote from: scottstancil
Would the ntpl only matter for a 2.4 kernel installation?

Don't think so. My Gentoo installs aren't quite "by-the-book", though. Basically, I start the smp kernel from the liveCD so as to get a 2.6 kernel (this may be moot in 2004.2, I haven't checked). Then when I'm chrooted and emerge system, I ax the kernel headers (they were 2.4) and rebuild my toolchain so as to get native thread support in stuff like gcc and glibc.

Now, I didn't just come up with all of that. I found a mini how-to on the gentoo forums that said doing it this way gave you a slight edge in performance. I'm a sucker for that sort of thing...

Oh, and sorry for the type-o. Its NPTL (Native Posix Thread Library). I do know that is significantly faster than pthreads.
