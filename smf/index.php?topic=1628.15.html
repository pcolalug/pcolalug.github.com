Slack should have a package you can install for the new modutils. Don't know a thing about Slack's package management, though, so hopefully, lloydr, imnes, danc, or one of our other Slack'ers can help you there. Barring that, you can get them from kernel.org and build from source (they should have detailed instructions on how to do that as well).

As for loading modules automagically at startup, the following is from the Gentoo Handbook, but should work the same on Slack:
Quote from: Gentoo handbook

Configuring the Modules

You should list the modules you want automatically loaded in /etc/modules.autoload.d/kernel-2.4 (or kernel-2.6). You can add extra options to the modules too if you want.

To view all available modules, run the following find command. Don't forget to substitute "<kernel version>" with the version of the kernel you just compiled:

Code Listing 21: Viewing all available modules
 
# find /lib/modules/<kernel version>/ -type f -iname '*.o' -or -iname '*.ko'

 

For instance, to automatically load the 3c59x.o module, edit the kernel-2.4 or kernel-2.6 file and enter the module name in it.

Code Listing 22: Editing /etc/modules.autoload.d/kernel-2.4
 
(Example for 2.4 kernels)
# nano -w /etc/modules.autoload.d/kernel-2.4

 
Code Listing 23: /etc/modules.autoload.d/kernel-2.4 or kernel-2.6
 
3c59x

 

Now run modules-update to commit your changes to the /etc/modules.conf file:


Finally, as far as I can tell, these problems you are having don't stem from the hardware so much as it does from the new kernel and non-open drivers (NVidia's license for the drivers is incompatable with the GPL - there's source code for the drivers in the download, though), so they aren't supported by the kernel developers. Its up to NVidia to make sure their drivers are compatable with the kernel. For the most part they've done a good job, but 2.6 development is hard to keep pace with these days, and there have been major changes from the 2.4 series. If its any consolation, it took me a week or so to get my nvidia card working and our mutual experience is not uncommon. These problems will go away as development of the 2.6 branch settles down.

after cd to the kernel source dir inside /usr/src/
i generally do:
1) make menuconfig (or make xconfig) whichever you choose to use(just avoid make config ROFL unless you wanna spend 4 years answering y(es)/n(o)/m(odule) questions)
2) make clean if needed
3) make bzImage
4) make modules
5) make modules_install
6) cp System.map to /boot
7) cp ./arch/i386/boot/bzImage to /boot named like vmlinuz-2.6.x (x being whatever version 2.6 kernel)
8) edit /etc/lilo.conf adding in the new option and leaving the old kernel's boot options in as backups
9) run the lilo command to update lilo and let it check syntax for any stupid mistakes
10) reboot

and after i reboot since i don't start X11 on boot it brings me to my usual terminal login in multiuser mode (default runlevel 3) because i hate x starting on boot
login as my user, cd into the directory where I have my nvidia driver install saved at which is generally in my /pub partition, su into root, sh NVIDIA*, since i deleted the old 4496 installer the 5336 loads, goes through it's module compile for the nvidia modules, loads them into my current kernel's module directory, logout of the root i sued into, startx as my user and it works.....never had a problem with this order not working on slackware with 2.6.x kernel and nvidia drivers

the old 4496 drivers you have to extract and patch for 2.6 kernel support, then just run make install and it loads the modules up for you...and with some via chipset boards and a few other peices i've heard you have to patch the 5336 set

im not sure i've ran into the missing agpgart problem but it's worth checking your config and make sure you didn't miss adding in support for something on your board....or try and load the nvidia driver and see if you run into any problems with x11 starting

also, check http://www.minion.de for any news on your hardware in specific having a problem, usually they have patches or tips for fixes on the site to help you get nvidia drivers working..... that site has helped me a lot back in the nvidia 4496 days after i already fully jumped into 2.6.0 and 2.6.0test# head first with no coming back lol

minion.de also has a few notes about things in the kernel that need to be disabled that are known to cause problems with nvidia drivers such as the CONFIG_4KSTACKS option in 2.6.6

EDIT: just came across something while skimming through minion.de:
" Linux 2.6 AGPGART seems to be broken on some chipsets. If you find that your system hangs upon starting X, potentially with ASCII garbage all over the screen, try the built-in NVIDIA AGP GART driver (Option "NvAgp" "1", AGPGART not loaded) instead. "
