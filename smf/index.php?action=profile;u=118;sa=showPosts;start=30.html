32  Distributions / Other Linux Distributions / Re: slack 11 xfs on sata drive on: October 28, 2006, 11:34:57 am
actually, for kernels, the xfs kernel DOES infact support sata.

http://slackware.mirrors.tds.net/pub/slackware/slackware-11.0/kernels/SATA-SUPPORT.TXT
Code:
xfs.s/config:# CONFIG_BLK_DEV_IDE_SATA is not set
xfs.s/config:CONFIG_SCSI_SATA=y
xfs.s/config:CONFIG_SCSI_SATA_AHCI=m
xfs.s/config:CONFIG_SCSI_SATA_SVW=m
xfs.s/config:CONFIG_SCSI_SATA_NV=m
xfs.s/config:CONFIG_SCSI_SATA_QSTOR=m
xfs.s/config:CONFIG_SCSI_SATA_PROMISE=m
xfs.s/config:CONFIG_SCSI_SATA_SX4=m
xfs.s/config:CONFIG_SCSI_SATA_SIL=m
xfs.s/config:CONFIG_SCSI_SATA_SIS=m
xfs.s/config:CONFIG_SCSI_SATA_ULI=m
xfs.s/config:CONFIG_SCSI_SATA_VIA=m
xfs.s/config:CONFIG_SCSI_SATA_VITESSE=m

it is set as modules as far as the individual devices, like all kernels. only difference in the sata kernel is that the devices are all set to yes.

also, in the sata kernel, xfs support is set as a module.

now, there is a tool called minitrd that is used in this situation

when you read /boot/README.initrd, you see this:
Quote
The usual reason to use an initrd is because you need to load kernel
modules before mounting the root partition. Usually these modules are
required to support the filesystem used by the root partition (ext3,
reiserfs, xfs), or perhaps the controller that the hard drive is attached
to (SCSI, RAID, etc). Essentially, there are so many different options
available in modern Linux kernels that it isn't practical to try to ship
many different kernels to try to cover everyone's needs. It's a lot more
flexible to ship a generic kernel and a set of kernel modules for it.

best bet would be use sata kernel and use mkinitrd for xfs, _OR_ if you look at using huge26 kernel..... it has _EVERYTHING_
Code:
huge26.s/config:# CONFIG_BLK_DEV_IDE_SATA is not set
huge26.s/config:CONFIG_SCSI_SATA=y
huge26.s/config:CONFIG_SCSI_SATA_AHCI=y
huge26.s/config:CONFIG_SCSI_SATA_SVW=y
huge26.s/config:CONFIG_SCSI_SATA_MV=y
huge26.s/config:CONFIG_SCSI_SATA_NV=y
huge26.s/config:CONFIG_SCSI_SATA_QSTOR=y
huge26.s/config:CONFIG_SCSI_SATA_PROMISE=y
huge26.s/config:CONFIG_SCSI_SATA_SX4=y
huge26.s/config:CONFIG_SCSI_SATA_SIL=y
huge26.s/config:CONFIG_SCSI_SATA_SIL24=y
huge26.s/config:CONFIG_SCSI_SATA_SIS=y
huge26.s/config:CONFIG_SCSI_SATA_ULI=y
huge26.s/config:CONFIG_SCSI_SATA_VIA=y
huge26.s/config:CONFIG_SCSI_SATA_VITESSE=y
huge26.s/config:CONFIG_SCSI_SATA_INTEL_COMBINED=y
all sata enabled, and, http://slackware.mirrors.tds.net/pub/slackware/slackware-11.0/kernels/huge26.s/config
looking at config, xfs is added directly in, rather than a module.
Code:
CONFIG_XFS_FS=y
CONFIG_XFS_EXPORT=y
CONFIG_XFS_QUOTA=y
CONFIG_XFS_SECURITY=y
CONFIG_XFS_POSIX_ACL=y

next time, I'd recommend researching the kernel variations first heh... they all have the same essential base. although, to save from having a monster bloated kernel with everything enabled like huge26, they just kick certain things back to module, but mkinitrd is always there.....

34  Club Stuff / Introductions / Re: Hello to the group on: October 09, 2006, 07:21:31 pm
Quote from: mobilemail on October 09, 2006, 06:48:18 pm
lloyd,
verizon xv6600  1xrtt  (I could do tethered with cable or bluetooth), for data access.

hmm from what I hear getting it working with bluetooth is easy as long as you have bluetooth pairing working. I don't have a bluetooth adapter so just used the usb cable

here is my post on sprintusers forum that shows what I did to get it working using usb cable:
Quote
current kernel version I'm running is 2.6.15.7 due to crappy ati drivers in the case of my laptop. But, built custom kernel for laptop and made sure to include USB ACM modem support (CONFIG_USB_ACM). with this and using the usb data cable that came with my A920, the phone shows up as /dev/ttyACM0 under dmesg.

then setup the PPP by creating these 2 files:

/etc/ppp/sprint.modem:
TIMEOUT 5
ABORT '\nBUSY\r'
ABORT '\nERROR\r'
ABORT '\nNO ANSWER\r'
ABORT '\nNO CARRIER\r'
ABORT '\nNO DIALTONE\r'
ABORT '\nRINGING\r\n\rRINGING\r'
'' \rAT
TIMEOUT 12
OK ATD#777
TIMEOUT 22
CONNECT ""

/etc/ppp/peers/sprint:
noauth
connect '/usr/sbin/chat -v -f /etc/ppp/sprint.modem'
defaultroute
usepeerdns
/dev/ttyACM0
921600
local
novj

Then, I simply ran the command "pppd call sprint" and watched my /var/log/messages for the modem status..... and at first, all was well, until after 2.5 minutes later, it'd error out and disconnect with the message of no response to 4 echo-requests.....

soooo, to fix that, I added these 2 lines to my /etc/ppp/peers/sprint:
lcp-echo-failure 4
lcp-echo-interval 65535

and everything works flawlessly now. after fixing the echo-request issue, I stayed connected for around 30minutes before I disconnected the PPP session myself since I was done.

this works for both 1xRTT and EVDO in the case of my samsung A920. in your case I'm not sure if you will use the same #777 for data call out to use the phone as a modem. and in the case of my phone, I had to disable modem NAI since it's an EVDO phone, to keep it from showing that I'm using it as a modem and not web useage just from phone (don't have tether web plan heh)
